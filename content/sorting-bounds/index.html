Comparisons and information theory: why the worst-case complexity of sorting is n log n

Posted 2020-06-19

In this post, we'll talk a bit about the usual proofs about the worst-case query complexity of sorting (at least, in the deterministic case) and then use a beautiful (and surprisingly simple!) tool from computational lower bounds to give a very general argument about the construction

The usual approach

There are many notes and posts talking about the fact that sorting will, in the worst case, always take Ω(nlog⁡n)\Omega(n \log n)Ω(nlogn) comparisons; equivalently, in the worst case, the number of comparisons is about a constant factor away from nlog⁡nn \log nnlogn when nnn is large. In many cases, the proof presented depends on the fact that the sorting algorithm is deterministic and goes a little like this (see here for a common case):

Preliminaries

Let CkC_kCk​ be the kkkth step of a deterministic sorting algorithm (this can be represented, for example, as a tuple containing what the next comparison should be) with input L∈KnL \in K^nL∈Kn where LLL is a list of nnn comparable elements. (For example, LLL can be a list of real numbers, in which case K=RK = \mathbb{R}K=R.)

By the definition of a deterministic algorithm, CkC_kCk​ depends only on the past k−1k-1k−1 comparisons; i.e., Ck(Ck−1,Ck−2,…,C1)C_k(C_{k-1}, C_{k-2}, \dots, C_1)Ck​(Ck−1​,Ck−2​,…,C1​). (I am slightly overloading notation here, of course, but the meaning should be clear.) This means that we can view the behavior of the algorithm as a tree, where CkC_kCk​ is a child node of Ck−1C_{k-1}Ck−1​ which itself is a child node of Ck−2C_{k-2}Ck−2​, etc. Additionally, the tree is binary since the output of a comparison is only one of two possibilities (if Ck=(a,b)C_k = (a, b)Ck​=(a,b), then either a≤ba\le ba≤b or b≤ab \le ab≤a).

Finally, let the leaf nodes of this tree be the list of indices (say, ppp, where each entry is an index, pi∈{1,…,n}p_i \in \{1, \dots, n\}pi​∈{1,…,n} for i=1,…,ni=1, \dots, ni=1,…,n) such that the list permuted at these indices is sorted, Lp1≤Lp2≤…,≤LpnL_{p_1} \le L_{p_2} \le \dots, \le L_{p_n}Lp1​​≤Lp2​​≤…,≤Lpn​​. Note that the number of nodes needed to get from the root node to a given leaf node (or permutation) is exactly the number of comparisons that the algorithm makes before it returns a specific permutation. If we can show that the height (the length of the longest path from root to leaf) of the tree is always larger than about nlog⁡nn \log nnlogn, then we've shown that this algorithm must take at least nlog⁡nn \log nnlogn steps.

Bound on the height

The idea for the bound is pretty simple: since this algorithm is a sorting algorithm and it can receive any unsorted list, then each of the n!n!n! possible permutations must be a leaf node of the tree. (Why?) Additionally, the maximum number of leaves for a binary tree of height hhh is 2h2^h2h, which, in turn, implies that we must have 2h≥n!2^h \ge n!2h≥n!. Taking the log of both sides shows that:

h=log⁡(2h)≥log⁡(n!)=Ω(nlog⁡n), h = \log(2^h) \ge \log(n!) = \Omega(n\log n), h=log(2h)≥log(n!)=Ω(nlogn),

which is exactly what we wanted to show. (The second equality is shown, for example, in the original reference for this proof; see page 2.)

Some takeaways from this proof

To prove this statement we only really used a few things: (a) the algorithm has to decide between n!n!n! things, and (b) at each query, it only receives a "yes" or a "no" (as we will soon make rigorous, it only gains 1 bit of information from each query). The rest of the proof simply sets up scaffolding for the remaining parts, most of which is really somewhat orthogonal to our intuition. The point is: look, we have n!n!n! things we have to decide on and every time we ask a question, we cut down our list of possible true answers by about half. How many times do we need to cut down our list to be able to have exactly one possible answer? (Of course, as we showed before, this should be log⁡2(n!)\log_2(n!)log2​(n!).)

Now, a good number of writings (and some textbooks) I've seen simply end here by saying "the algorithm gains at most 1 bit for every query and we need at least log⁡(n!)∼nlog⁡n\log(n!) \sim n \log nlog(n!)∼nlogn bits" and then give some vague citation to Shannon's theorem about communication without explicitly writing out any argument. (While there is some connection, it's unclear what it is or how to even make it rigorous in a general sense.) It's a bit of a shame since it doesn't really take much math to fully justify this intuitive statement, as we will see next.

An information-theoretic approach

The idea behind a (simple) information theoretic approach is to view the algorithm as attempting to 'uncover' the true, sorted permutation by querying an oracle that knows whether two elements are in sorted order or not. Here, the oracle gets to choose some 'true' permutation uniformly at random and then the sorting algorithm queries the oracle in a restricted sense: it can only ask yes or no questions about the list.

More formally, we will let X\mathcal XX be the set of all n!n!n! possible permutations of 111 through nnn and let XXX the oracle's permutation such that X∼XX \sim \mathcal XX∼X is uniformly randomly sampled from X\mathcal XX. Then, the algorithm gets to ask a sequence of yes/no queries Yi∈{0,1}Y_i \in \{0, 1\}Yi​∈{0,1} for i=1,…,ki=1, \dots, ki=1,…,k which are dependent on the permutation XXX, and, at the end of kkk queries, must respond with some 'guess' for the true permutation X^\hat XX^, which is a random variable that depends only on YYY.

We can represent the current set up as a Markov chain X→Y→X^X \to Y \to \hat XX→Y→X^, since X^\hat XX^ is conditionally independent of XXX given YYY (i.e., the algorithm can only use information given by YYY) while the random variable YYY depends only on XXX. The idea then is to give a bound on the number of queries kkk required for an algorithm to succeed with probability 1.

To give a lower bound on this quantity, we'll use a tool from information theory called Fano's inequality, which, surprisingly, I don't often see taught in information theory courses. (Perhaps I haven't been taking the right ones!)

Fano's inequality

I learned about this lovely inequality in John Duchi's class, EE377: Information Theory and Statistics (the lectures notes for the class are here). Its proof really makes it clear why entropy, as defined by Shannon, is pretty much the exact right quantity to look at. We'll explore a weaker version of it here that is simpler to prove and requires fewer definitions but which will suffice for our purposes.

We will use X,YX, YX,Y as random variables and set HHH as the entropy, defined:

H(X)=−∑X∈XP(X)log⁡P(X), H(X) = -\sum_{X \in \mathcal{X}}P(X) \log P(X), H(X)=−X∈X∑​P(X)logP(X),

where X\mathcal{X}X is the space of values that XXX can take on. The conditional entropy of XXX given YYY is defined as

H(X∣Y)=−∑Y∈YP(Y)∑X∈XP(X∣Y)log⁡P(X∣Y)=H(X,Y)−H(Y). H(X|Y) = -\sum_{Y \in \mathcal Y} P(Y) \sum_{X \in \mathcal X} P(X\mid Y) \log P(X\mid Y) = H(X, Y) - H(Y). H(X∣Y)=−Y∈Y∑​P(Y)X∈X∑​P(X∣Y)logP(X∣Y)=H(X,Y)−H(Y).

As usual, the entropy HHH is a measure of the 'uncertainty' in the variable XXX, with the maximally uncertain distribution being the uniform one.[1] Additionally, note that H(X,Y)H(X, Y)H(X,Y) is the entropy taken with respect to the joint distribution of XXX and YYY. Finally, if XXX is a deterministic function of YYY, then H(X∣Y)=0H(X\mid Y) = 0H(X∣Y)=0 which follows from the definition.

For this post, we will only make use of the following three properties of the entropy (I will not prove them here, as there are many available proofs of them, including the notes above and Cover's famous Elements of Information Theory):

  1. The entropy can only decrease when removing variables, H(X,Y)≥H(X)H(X, Y) \ge H(X)H(X,Y)≥H(X).

  2. The entropy is smaller than the log of the size of the sample space H(X∣Y)≤H(X)≤log⁡∣X∣H(X \mid Y) \le H(X) \le \log|\mathcal X|H(X∣Y)≤H(X)≤log∣X∣. (Equivalently, conditioning reduces entropy, and the uniform distribution on X\mathcal XX has the highest possible entropy, log⁡∣X∣\log|\mathcal X|log∣X∣.)

  3. If a random variable X^\hat XX^ is conditionally independent of XXX given YYY (i.e., if X→Y→X^X \to Y \to \hat XX→Y→X^ is a Markov chain), then H(X∣X^)≥H(X∣Y)H(X\mid \hat X) \ge H(X\mid Y)H(X∣X^)≥H(X∣Y). This is often called a data processing inequality, which simply says that XXX has smaller entropy knowing YYY than knowing a variable, X^\hat XX^, that has undergone further processing. In other words, you cannot gain more information about XXX from a variable YYY by further processing YYY.

This is all we need to prove the following inequality. Let X→Y→X^X \to Y \to \hat XX→Y→X^ be a Markov chain such that X^\hat XX^ is conditionally independent of XXX given YYY and XXX is uniformly drawn from X\mathcal XX, then the probability that X≠X^X \ne \hat XX=X^ is given by PeP_ePe​ and PeP_ePe​ satisfies

Pe≥1−k+1log⁡∣X∣, P_e \ge 1 - \frac{k+1}{\log |\mathcal X|}, Pe​≥1−log∣X∣k+1​,

where kkk is the number of binary queries made and ∣X∣|\mathcal X|∣X∣ is the number of elements in X\mathcal XX.

A quick aside

Proving this inequality is enough to show the claim. Note that, we want the probability of error to be 0 (since we want our algorithm to work!) so

1−k+1log⁡∣X∣≤0impliesk+1≥log⁡∣X∣, 1 - \frac{k+1}{\log |\mathcal X|} \le 0 \quad \text{implies} \quad k+1 \ge \log |\mathcal X|, 1−log∣X∣k+1​≤0impliesk+1≥log∣X∣,

and, since ∣X∣|\mathcal X|∣X∣ is the space of possible permutations of size nnn, (of which there are n!n!n! of) then ∣X∣=n!|\mathcal X| = n!∣X∣=n! and log⁡n!∼Ω(nlog⁡n)\log n! \sim \Omega(n \log n)logn!∼Ω(nlogn) then kkk must satisfy

k=Ω(nlog⁡n). k = \Omega(n \log n). k=Ω(nlogn).

In other words, the number of queries (or comparisons, or whatever) kkk must be approximately at least as large as nlog⁡nn \log nnlogn, asymptotically (up to constant multiples). In fact, this proves a slightly stronger statement that no probabilistic algorithm can succeed with nonvanishing probability if the number of queries is not on the order of nlog⁡nn \log nnlogn, which our original proof above does not cover!

Now, the last missing piece is showing that the probability of error is bounded from below.

A proof of Fano's inequality

This is a slightly simplified version of the proof presented in Duchi's notes (see section 2.3.2) for the specific case we care about, which requires less additional definitions and supporting statements. Let EEE be the 'error' random variable that is 111 if X≠X^X \ne \hat XX=X^ and 0 otherwise, then let's look at the quantity H(E,X∣X^)H(E, X\mid \hat X)H(E,X∣X^):

H(E,X∣X^)=H(X∣X^,E)+H(E∣X^), H(E, X\mid \hat X) = H(X\mid \hat X, E) + H(E\mid \hat X), H(E,X∣X^)=H(X∣X^,E)+H(E∣X^),

by our definition of conditional entropy, and

H(X∣X^,E)=PeH(X∣X^,E=1)+(1−Pe)H(X∣X^,E=0), H(X\mid \hat X, E) = P_e H(X\mid \hat X, E=1) + (1-P_e)H(X\mid \hat X, E=0), H(X∣X^,E)=Pe​H(X∣X^,E=1)+(1−Pe​)H(X∣X^,E=0),

again by the same definition. Since X=X^X = \hat XX=X^ whenever there is no error, E=0E=0E=0, then H(X∣X^,E=0)=0H(X\mid \hat X, E=0) = 0H(X∣X^,E=0)=0 since XXX is known, so, we really have

H(E,X∣X^)=PeH(X∣X^,E=1)+H(E∣X^). H(E, X\mid \hat X) = P_e H(X\mid \hat X, E=1) + H(E\mid \hat X). H(E,X∣X^)=Pe​H(X∣X^,E=1)+H(E∣X^).

Since EEE can only take on two values, we have that H(E∣X^)≤log⁡(2)=1H(E\mid \hat X) \le \log(2) = 1H(E∣X^)≤log(2)=1 and we also have that H(X∣X^,E=1)≤log⁡∣X∣H(X\mid \hat X, E=1) \le \log |\mathcal X|H(X∣X^,E=1)≤log∣X∣, which gives

H(E,X∣X^)≤Pelog⁡∣X∣+1. H(E, X\mid \hat X) \le P_e \log|\mathcal X| + 1. H(E,X∣X^)≤Pe​log∣X∣+1.

Now, we have that

H(E,X∣X^)≥H(X∣X^)≥H(X∣Y). H(E, X\mid \hat X) \ge H(X\mid \hat X) \ge H(X\mid Y). H(E,X∣X^)≥H(X∣X^)≥H(X∣Y).

The first inequality follows from the fact that we're removing a variable and the second follows from statement 3 in the previous section (as X→Y→X^X \to Y \to \hat XX→Y→X^). Using the definition of H(X∣Y)H(X\mid Y)H(X∣Y), then we have

H(X,Y)−H(Y)≥H(X)−H(Y)=log⁡∣X∣−H(Y)≥log⁡∣X∣−log⁡∣Y∣. H(X, Y) - H(Y) \ge H(X) - H(Y) = \log |\mathcal X| - H(Y) \ge \log |\mathcal X| - \log|\mathcal Y|. H(X,Y)−H(Y)≥H(X)−H(Y)=log∣X∣−H(Y)≥log∣X∣−log∣Y∣.

The first inequality here follows since we're (again!) removing a variable and the equality follows from the fact that XXX is uniformly randomly drawn from X\mathcal XX and the last inequality follows from the fact that the entropy of YYY is always smaller than that of the uniform distribution on Y\mathcal YY. Finally, note that, if we have kkk queries, then ∣Y∣=2k|\mathcal Y| = 2^k∣Y∣=2k (this is the number of possible values a sequence of kkk binary queries can take on). So, log⁡∣Y∣=k\log |\mathcal Y| = klog∣Y∣=k (in other words, the maximum amount of information we can get with kkk binary queries is kkk bits) so we find

log⁡∣X∣−k≤Pelog⁡∣X∣+1, \log |\mathcal X| - k \le P_e\log |\mathcal X| + 1, log∣X∣−k≤Pe​log∣X∣+1,

or, after some slight rewriting:

Pe≥1−k+1log⁡∣X∣, P_e \ge 1 - \frac{k+1}{\log |\mathcal X|}, Pe​≥1−log∣X∣k+1​,

as required!

Some discussion

I think overall that Fano's inequality is a relatively straightforward way of justifying a ton of the statements one can make about information theory without needing to invoke a large number of more complicated notions. Additionally, the proof is relatively straightforward (in the sense that it only requires very few definitions and properties of the entropy) while also matching our intuition about these problems pretty much exactly.

In particular, we see that sorting is hard not because somehow ordering elements is difficult, but because we have to decide between a bunch of different items (in fact, n!n!n! of them) while only receiving a few bits of information at any point in time! In fact, this bound applies to any yes/no query that one can make of the sorted data, not just comparisons, which is interesting.

There are some even more powerful generalizations of Fano's inequality which can also be extended to machine learning applications: you can use them to show that, given only some small amount of data, you cannot decide between a parameter that correctly describe the data and one that does not.

This is all to say that, even though entropy is a magical quantity, that doesn't mean we can't say very rigorous things about it (and make our intuitions about lower bounds even more rigorous, to boot!).

[1] In fact, the entropy is really a measure of how close a variable is to the uniform distribution, in the case of compact domains X\mathcal XX—the higher the entropy, the closer it is.
